# ルーティング（AppRouter構成）非対応の決定

このドキュメントは [Issue #4「#004リファクタリング実施」](https://github.com/Yaeshio/SQL-Viz/issues/4)
で検討した「ルーティングをどう扱うか」の決定と、その理由を記録するものである。
[user-stories.md](./user-stories.md) と同様、決定の「なぜ」を残す目的で新設した。

## 1. 背景

Issue #4 本文には以下の一文がある。

> また、現在、Tailwindcssもインストールされているため、AppRouter構成として
> 適切なルーティングを検討するべきである。

一方で、[README.md](../README.md) や
[Sql animation tool spec .md](./Sql%20animation%20tool%20spec%20.md) を含む
本プロジェクトのどのドキュメントにも、SQLエディタ＋キャンバスの単一画面
以外のページ構想は存在しない。

## 2. 検討した選択肢

- **react-router 等を導入し、複数ページ構成にする** — 現時点で具体的に
  追加したいページ（設定画面・履歴画面など）が存在せず、導入しても
  単一ルートを定義するだけの空箱になる。
- **react-router を導入し、土台だけ用意しておく** — 将来のページ追加を
  見越した先行投資だが、「明確な必要性がない限り新たなパッケージを
  追加しない」という [CLAUDE.md](../CLAUDE.md) のデザイン方針に反する。
- **導入を見送る（採用）** — 具体的な複数画面のニーズが生まれた時点で
  改めて検討する。

## 3. 決定

**ルーティングの導入は見送る。** 新規依存（`react-router-dom` 等）は
追加しない。`App.tsx` は引き続き単一の構成ルートとして残る。

理由は、YAGNI（You Aren't Gonna Need It）原則に従い、必要性が具体化
していない抽象化を先取りしないこと。本アプリはバックエンド・永続化を
持たないクライアントのみの単一画面プロトタイプであり、ルーティングを
導入しても解決する問題が現時点で存在しない。

## 4. 再検討条件

以下のような、具体的に複数画面が必要になる要求が生まれた時点で改めて
検討する。

- SQLエディタ＋キャンバス以外の独立した画面（例: 設定画面、実行履歴
  画面、ヘルプ/オンボーディング画面）を追加する要求が出た場合
- 現在の単一画面内の状態（エディタの内容・実行ログ等）をURLで共有・
  ブックマーク可能にしたいという要求が出た場合

## 5. 影響

- `package.json` の依存関係に変更なし
- `App.tsx` の構成（`main.tsx` から直接マウントされるエントリコンポー
  ネント）に変更なし
