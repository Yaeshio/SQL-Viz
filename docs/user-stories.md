# ユーザーストーリー

このドキュメントは Issue #2「スモークテスト整備」のフォローアップコメントに基づき、
SQL-Viz が表現する SQL 文を**なぜその範囲に選定したか**、**誰が何をしたくてその構文を
使うのか**を明文化するものである。[smoke-test-spec.md](./smoke-test-spec.md) が
「何を・なぜテストするか」を、[smoke-test-design.md](./smoke-test-design.md) が
「どう実装するか」を担うのに対し、本書はそのさらに上流、スモークテストの各シナリオが
実際の利用者の目的に基づいていることの裏付けを担う。

## 1. 背景

Issue #2 には次のフォローアップコメントがある。

> すべてが1つのゴールデンサンプルを中心に生成されている。App.tsx の SAMPLE
> （CREATE→INSERT→WHERE id > 1）だけが end-to-end で通ればよい、という形で
> 足場が組まれています。サンプルが通る分岐は本物、通らない分岐（複合WHERE・LIKE・
> 未知列）は「それっぽい no-op」で埋まる。…このような状態でスモークテストを
> 作成するのではなく、まずはこのシステムで表現する SQL 文を選定し、これに対する
> ユーザーストーリーを書き出して、仕様書を更新し、その後スモークテストを作成する
> べきではないだろうか。

このコメント時点で懸念されていた「未対応の句が黙って no-op になる」非対称性は
[Issue #3](https://github.com/Yaeshio/SQL-Viz/issues/3)（`parser.ts` の許可リスト化）
により技術的には解消済みである。しかしコメントが求めていた「SQL 文の選定」と
「ユーザーストーリー」はまだどこにも存在しなかった。本書はその2点を埋める。

## 2. 想定ユーザー像（ペルソナ）

[README.md](../README.md) と [Sql animation tool spec .md](./Sql%20animation%20tool%20spec%20.md)
に書かれている製品意図（実DBへの接続を行わない、クライアントのみのSQL可視化
プロトタイプ）から導ける範囲で、以下の2つのペルソナを想定する。詳細な業務文脈は
作らず、コード・README から裏付けられる範囲に留める。

- **P1: SQL学習者** — `CREATE TABLE`/`INSERT`/`SELECT` の基本を学び始めたばかりで、
  SQL文を実行した結果テーブルやデータがどう変化するかを、文字の羅列ではなく
  目で見て理解したい人。
- **P2: SQLをデモする人** — チームや授業で、SQL文がテーブルの状態にどう影響するか
  （行の追加、`WHERE`によるフィルタとその解除など）を対面で見せたい人。

## 3. 選定されたSQL文一覧（対応範囲の追認）

対応/非対応の具体的な構文一覧は [smoke-test-spec.md](./smoke-test-spec.md) 4節、
および [README.md](../README.md) の「できること」節に技術的な正確さを保って
記載されている。本節ではそれを重複して列挙せず、**この範囲が実装の結果ではなく、
上記ペルソナのゴールデンパスを満たすために意図的に選定されたものである**ことを
明記する。

選定された対応範囲（Phase 1 MVP、[Sql animation tool spec .md](./Sql%20animation%20tool%20spec%20.md)
7節と対応）:

- `CREATE TABLE table (col type, ...)`（単一テーブル、列定義のみ）
- `INSERT INTO table [(cols)] VALUES (...)`（単一行・複数行）
- `SELECT * | cols FROM table [WHERE col op value]`（単一の二項比較のみ）

これは P1・P2 に共通する最小のゴールデンパス「`CREATE TABLE` でテーブルを作り、
`INSERT` でデータを入れ、`SELECT ... WHERE` で絞り込む」を一気通貫で見せられる
最小構成であり、それ以外（`JOIN`・複合`WHERE`・`UPDATE`/`DELETE`等）は、
中途半端に対応して誤った可視化結果を見せるよりも、明示的に「対応していない」と
伝えることを優先し、今回は選定範囲に含めない。

## 4. 対応SQL文ごとのユーザーストーリー

| ID | ユーザーストーリー |
|---|---|
| US-01 | P1として、`CREATE TABLE` を実行してテーブルがキャンバス上に出現する様子を見たい。なぜなら、テーブルという概念が「文字」ではなく「実体」として作られることを直感的に理解したいから。 |
| US-02 | P1として、`INSERT INTO ... VALUES (...)` を複数回（または複数行）実行して、行が1件ずつ増えていく様子を見たい。なぜなら、`INSERT` の実行順序とデータの蓄積過程を確認したいから。 |
| US-03 | P1として、`SELECT * FROM table`（`WHERE`なし）を実行して、テーブルの現在の全データを一覧で確認したい。なぜなら、`WHERE`を使う前提として「今何が入っているか」を把握したいから。 |
| US-04 | P1として、`SELECT col FROM table WHERE col op value` を実行して、条件に合わない行がフェードアウトし、合う行だけが残る様子を見たい。なぜなら、`WHERE`句が「行を絞り込む」ことを、削除ではなく一時的なフィルタとして視覚的に理解したいから。 |
| US-05 | P1として、`WHERE`付きの`SELECT`を実行した直後に`WHERE`なしで`SELECT`を実行し、一度フェードアウトした行が再び現れる様子を見たい。なぜなら、`WHERE`によるフィルタが行を破壊するものではなく可逆的であることを確認したいから。 |
| US-06 | P2として、複数の`CREATE TABLE`を続けて実行し、複数のテーブルがキャンバス上で重ならずに配置される様子をデモしたい。なぜなら、複数テーブルを扱うSQLの説明において、レイアウトの破綻が説明の妨げにならないようにしたいから。 |

## 5. 非対応SQL文に対するユーザーストーリー

| ID | ユーザーストーリー |
|---|---|
| US-07 | P1として、まだツールが対応していない構文（`UPDATE`/`DELETE`/`JOIN`/複合`WHERE`等）を書いてしまったときに、黙って不完全な結果を見せられるのではなく、はっきりと「この構文には対応していない」というエラーメッセージを見たい。なぜなら、誤った可視化結果を正しいものと誤解したまま学習してしまうことを避けたいから。 |

この US-07 が、`parser.ts` の許可リスト方式（Issue #3）による
`Unsupported statement type: ...` / `Unsupported clause: ...` エラーの
存在理由である。個々の非対応構文（`UPDATE`、`JOIN`、複合`WHERE`など）に対して
個別のユーザーストーリーは設けず、この1つのストーリーがすべてに共通する
根拠として機能する。

## 6. `SMOKE-*` シナリオとの対応表

[smoke-test-spec.md](./smoke-test-spec.md) 5節の各シナリオが、どのユーザー
ストーリーに基づくかを以下に示す。

| SMOKE-* | 対応するUS | 備考 |
|---|---|---|
| SMOKE-01 | US-01, US-02, US-04 | ゴールデンシナリオそのもの |
| SMOKE-02 | US-06 | 複数テーブルのレイアウト |
| SMOKE-03 | US-03 | `SELECT *` |
| SMOKE-04 | US-05 | フィルタ解除 |
| SMOKE-05 | US-01, US-02 | 各データ型でのCREATE+INSERT |
| SMOKE-06 | — | 実装保護のためのエラーハンドリング（存在しないテーブル）。ユーザーの意図的操作というより誤操作時の防御であり、対応するUSなし |
| SMOKE-07 | — | 同上（テーブル名重複） |
| SMOKE-08 | — | 同上（列数不一致） |
| SMOKE-09 | — | 同上（構文エラー） |
| SMOKE-10 | — | 同上（複数文途中のエラー、早期終了の実装保護） |
| SMOKE-11〜13 | US-07 | `UPDATE`/`DELETE`/`ALTER TABLE` |
| SMOKE-14〜16 | US-07 | `JOIN`/複合`WHERE`/`GROUP BY`等 |

SMOKE-06〜10 に対応するUSが無いのは意図的である。これらは「ユーザーが達成したい
目的」ではなく、実装が壊れていないことを確認する防御的なエラーハンドリングの
検証であり、無理にユーザーストーリーへこじつけない。

## 7. 将来の拡張時の運用

新しいSQL構文（`UPDATE`/`JOIN`/複合`WHERE`等）に対応する際は、以下の順序で
進める。

1. 本書に該当する `US-*` を追加する（誰が何をしたいのか）
2. [Sql animation tool spec .md](./Sql%20animation%20tool%20spec%20.md)／
   [smoke-test-spec.md](./smoke-test-spec.md)／[test-spec.md](./test-spec.md)
   の対応SQL文一覧・シナリオ表を更新する（該当箇所を `Unsupported clause`
   検証から正常系シナリオ検証へ書き換える）
3. 対応するテストケース（`tests/*.test.ts`／`tests/smoke.test.ts`）を先に
   書く・更新する
4. 最後に `src/parser.ts`／`src/reducer.ts` 等のソースコードを実装する

この順序（ドキュメント → テスト → 実装）により、実装が先行してテストが
後追いで「それっぽく」つじつま合わせされることを防ぐ。本節は
[smoke-test-design.md](./smoke-test-design.md) 6節・[test-design.md](./test-design.md)
7節の「将来の拡張時のガイドライン」と対応関係にある。
